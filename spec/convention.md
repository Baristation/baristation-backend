# 🛠 Java 객체 생성 컨벤션 가이드

코드의 가독성, 유지보수성, 그리고 객체지향적 원칙을 지키기 위해 상황에 따라 **정적 팩토리 메서드**와 **빌더 패턴**을 구분하여 사용합니다.

---

## 📊 요약 및 비교

| 항목 | 정적 팩토리 메서드 (`from`, `of`) | 빌더 패턴 (`@Builder`) |
| :--- | :--- | :--- |
| **주 목적** | 데이터 타입 변환 및 캡슐화 | 데이터 단계적 조립 및 가독성 |
| **주요 재료** | 이미 완성된 객체 (Entity, Map) | 분산된 변수, 로직의 결과값들 |
| **의존성** | DTO가 원본 객체의 구조를 알고 있음 | 호출부에서 생성 과정을 제어함 |
| **권장 예시** | `BeanImageResponse.from(entity)` | `TokenResponse.builder()...build()` |

---

## 1. 빌더 패턴 (Builder Pattern)
> **사용 키워드**: `@Builder`

### ✅ 사용 상황: "데이터의 조립 및 직접 생성"
여러 소스에서 온 데이터를 **조합(Assembly)**하거나, 비즈니스 로직의 결과물을 바탕으로 **새로운 객체를 창조**할 때 사용합니다.

- **조합형 DTO**: 여러 로직의 결과물(예: AccessToken + RefreshToken)을 합쳐 응답할 때 (`TokenResponse`)
- **초기 엔티티 생성**: 클라이언트의 입력값을 모아 새로운 데이터(Entity)를 처음 만들 때
- **복잡한 생성**: 인자가 많아 순서가 헷갈리거나, 특정 필드만 선택적으로 채워야 할 때

### 💡 장점
- 인자 순서로 인한 휴먼 에러를 방지 (특히 동일 타입의 인자가 많을 때 필수)
- 각 필드에 어떤 값이 들어가는지 직관적으로 확인 가능 (가독성)
- 나중에 필드가 추가되거나 순서가 바뀌어도 기존 코드가 깨지지 않음

---

## 2. 정적 팩토리 메서드 (Static Factory Method)
> **사용 키워드**: `from()`, `of()`

### ✅ 사용 상황: "데이터의 변환 및 재사용"
이미 존재하는 객체(Entity, Map 등)를 다른 규격(DTO)으로 **포장(Wrapping)**하거나 **타입을 변환(Conversion)**할 때 사용합니다.

- **Entity → DTO 변환**: DB에서 조회한 데이터를 클라이언트에 전달할 때
- **외부 데이터 파싱**: 소셜 로그인(OAuth2) API 응답(Map)을 객체화할 때 (`KakaoUserInfoDTO` 등)
- **단순 객체 생성**: 생성자 인자가 1~2개로 매우 적고 의도가 명확할 때

### 💡 장점
- 메서드 이름을 통해 생성 목적을 명확히 드러낼 수 있음 (`fromEntity`, `ofAttributes`)
- 객체 생성 로직을 DTO 내부에 캡슐화하여 서비스 레이어의 코드가 간결해짐
- Java `record` 사용 시 불필요한 빌더 구현을 줄여 보일러플레이트 제거
