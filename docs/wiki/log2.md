# 🛡️ 백엔드 로그 보관 및 보안 정책 가이드 (Logging Storage & Security Guidelines)

본 가이드는 "실전 백엔드 로그 최적화 가이드"의 확장판으로, 시스템의 추적성을 보장하면서도 스토리지 비용을 최적화하고 보안 컴플라이언스를 준수하기 위한 세부 정책을 정의합니다.

---

## 1. 로그 보관 및 스토리지 (TTL & Storage Strategy)

로그는 시스템의 상태 파악을 위해 필수적이지만, 무한정 보관할 경우 스토리지 비용을 낭비하게 됩니다. 데이터의 활용 목적에 따라 Hot/Cold 스토리지로 분리하여 관리합니다.

* **Hot Storage (즉시 검색 가능 - ElasticSearch, CloudWatch 등):**
    * **보관 기간:** **7일 ~ 14일** (장애 발생 시 즉각적인 트러블슈팅 목적)
    * **대상:** `ERROR`, `WARN`, `INFO` 수준의 모든 애플리케이션 로그
* **Cold Storage (아카이빙 - AWS S3 등):**
    * **보관 기간:** 14일 이후 ~ 영구 (또는 법적 의무 보관 기간까지)
    * **대상:** 비즈니스 지표 분석이나 법적 증빙이 필요한 주요 API `INFO` 로그 및 `ERROR` 로그
    * **방식:** 날짜별 압축 파일(gzip 등) 형태로 저렴한 객체 스토리지에 보관

---

## 2. 로그 저장 방식 및 포맷 표준 (Logback 기준)

애플리케이션의 메인 스레드 성능에 영향을 주지 않고, 로그 수집기(Logstash, Fluentd 등)가 쉽게 파싱할 수 있도록 구성합니다.

* **RollingFileAppender 적용:** 일(Daily) 단위로 로그 파일을 분리하고, 파일 크기가 설정값(예: 100MB)을 초과하면 새로운 파일로 분리(Rolling)합니다.
* **비동기 로깅 (AsyncAppender):** 파일 I/O 대기 시간으로 인한 비즈니스 로직(JPA 트랜잭션 등)의 지연을 방지하기 위해 반드시 비동기 방식으로 로그를 기록합니다.
* **구조화된 JSON 포맷:** 검색과 분석을 용이하게 하기 위해 단순 텍스트가 아닌 JSON 포맷으로 출력합니다. (예: `LogstashEncoder` 활용)

## 3. 민감 정보 및 결제 데이터 로깅 수칙

보안 프레임워크(Spring Security, OAuth2 등)가 적용된 환경에서는 "무엇을 남길 것인가"보다 **"무엇을 남기지 않을 것인가"**가 훨씬 중요합니다.

### ① 절대 로깅 금지 (Zero-Logging) 대상
아래 항목은 어떠한 로그 레벨에서도 절대 출력되거나 저장소에 기록되어서는 안 됩니다.
* **인증/인가 정보:** JWT (Access Token, Refresh Token), OAuth2 인가 코드, 세션 ID
* **비밀번호:** 평문 및 해시(Bcrypt 등) 처리된 비밀번호
* **결제 핵심 정보:** 카드 번호 전체, CVC, 카드 유효기간 (PCI-DSS 규정 위반)

### ② 마스킹 (Masking) 필수 적용 대상
에러 추적을 위해 불가피하게 개인정보(PII)를 남겨야 할 경우, 반드시 마스킹 처리를 거쳐야 합니다.
* **개인정보:** 이메일(`s*******@gmail.com`), 이름(`최*우`), 전화번호, 주소 등
* **적용 방법:** 로깅 필터(Filter)나 인터셉터(Interceptor) 단에서 HTTP Body 내부의 민감한 JSON 필드를 정규식으로 치환하여 로깅합니다.

### ③ 결제/중요 트랜잭션 로깅 방법
결제와 같은 중요 비즈니스 로직은 상세한 데이터 대신 **식별자와 상태값** 위주로 기록합니다.
* **기록 항목:** 내부 주문 ID(Order ID), Trace-ID, 결제 상태 코드(SUCCESS/FAIL), 외부 PG사 승인 번호, 에러 코드
* **예시:** ```json
  {"level":"INFO", "type":"PAYMENT", "orderId":"B20260509-001", "status":"SUCCESS", "amount":5500, "traceId":"abc-123"}
    ```

## 4. Trace-ID 적용 표준 (MDC 활용)

흩어진 로그들을 하나의 요청 흐름으로 묶어주는 `Trace-ID`는 백엔드 디버깅의 핵심입니다.

* **생성 및 부여:** 클라이언트 요청이 최초 진입하는 지점(Filter 또는 Interceptor)에서 UUID를 생성합니다.
* **MDC (Mapped Diagnostic Context) 활용:** 생성된 UUID를 스레드 로컬에 기반한 `MDC.put("traceId", uuid)`로 저장하여, 이어지는 모든 서비스/레포지토리 계층의 로그(`%X{traceId}`)에 자동 포함되도록 합니다.
* **초기화 (Memory Leak 방지):** 요청 처리가 완료되어 응답이 나가는 시점에 반드시 `MDC.clear()`를 호출하여, 스레드 풀 재사용 시 발생하는 데이터 오염을 방지해야 합니다.
